# Prompt Improvement Changelog

Date: 2026-02-22
Branch: `copilot/audit-prompt-improvements`

---

## Summary

Every prompt file in `harness/prompts/` and every template in `harness/templates/` was audited
against the evaluation criteria in the problem statement: specificity, constraint completeness,
ownership alignment, structure, consistency, and redundancy. Changes were kept surgical—only files
with material defects were modified. The primary themes were: (1) fixing a silent but consequential
scene-count mismatch between `manifest.yaml` and the live user prompt; (2) removing a direct
contradiction between `harmonious_color` being listed as a usable helper and being explicitly
banned in the same files; (3) encoding missing non-negotiable constraints (no-placeholder narration,
timing ownership) into the prompts and shared rules so the pipeline can enforce them without relying
on implicit agent knowledge; and (4) adding ownership labels to template sections that described
script-owned tasks in a way that could mislead agents into attempting to perform them.

---

## File-by-File Changes

### `harness/prompts/00_plan/manifest.yaml`

**What changed:** Corrected scene count from `8-12` to `12-24` and total duration from `240-480 s`
to `480-960 s` in both the `inputs.constraints` field and the `validation` block.

**Why:** The `manifest.yaml` values conflicted with the live `user.md` (which said 12-24 / 480-960)
and the authoritative `docs/reference_docs/phase_plan.md` reference. This mismatch violated
**Criterion 5 (Consistency)** and could cause silent over-counting or under-counting of scenes if
any tooling reads the manifest for validation limits.

---

### `harness/prompts/00_plan/system.md`

**What changed:** Added a "Non-negotiable output constraints" block and a "Failure conditions"
block that explicitly list: no `id`/`narration_key` in scene objects; every `visual_ideas` entry
must name a Manim CE mobject class; a recap scene is required; and scene counts/durations must
be within the declared ranges.

**Why:** The original system prompt was entirely behavioral guidance with no explicit constraints
or failure conditions (**Criterion 1 – Specificity**, **Criterion 4 – Structure**). Agents could
produce technically valid JSON that still violated the pipeline contract without any guidance about
what constitutes a rejected response.

---

### `harness/prompts/01_review/manifest.yaml`

**What changed:** Removed the stale `scene_id(scene_XX_slug)` constraint (which doesn't match the
actual `scene_01`, `scene_02` ID format generated by the orchestrator). Added a comment
clarifying the stub status. Removed `review_notes.md` output fields since this phase is not
LLM-driven. Updated scene count constraint to match the corrected 12-24 value.

**Why:** The old manifest contained an incorrect ID format constraint (**Criterion 5 –
Consistency** with actual runtime behavior). Documenting a stub phase as having LLM output fields
is misleading (**Criterion 4 – Structure**).

---

### `harness/prompts/02_narration/system.md`

**What changed:** Added a "Non-negotiable output constraints" block encoding: (a) no
placeholder-only narration values (parser rejects them); (b) narration text is spoken by Qwen
TTS — write spoken sentences only; (c) never hardcode narration in scene files. Added a
"Failure conditions" block.

**Why:** The parser in `harness/parser.py` rejects placeholder-only strings (e.g., `"..."`) with
a validation error, but the prompt gave no guidance on this rule. Agents could produce invalid
narration without knowing why it would be rejected (**Criterion 2 – Constraint Completeness**,
**Criterion 4 – Structure**). The voice policy mention aligns the prompt with AGENTS.md §15.

---

### `harness/prompts/_shared/core_rules.md`

**What changed:** Added **Rule 8 – Timing Ownership** under the CRITICAL RULES section. The new
rule states: use `tracker.duration` proportional fractions only; never hardcode raw second values
that are not derived from `tracker.duration`; never compute cumulative timing sums by hand; use
`min()`/`max()` guards to bound individual values.

**Why:** `DEVELOPMENT_GUIDELINES.md` explicitly states timing arithmetic must be deterministic and
script-owned, and agents must not hand-author fragile timing arithmetic. This rule was documented
in `visual_helpers.md` as a one-liner comment but was absent from the shared core rules that apply
to all phases (**Criterion 2 – Constraint Completeness**, **Criterion 3 – Ownership Alignment**).

---

### `harness/prompts/INDEX.md`

**What changed:** Added the missing `training` phase row (`03_training`, active, readiness
handshake) and normalised column casing to lowercase for consistency.

**Why:** `03_training` has a full `system.md`/`user.md`/`manifest.yaml` set and is invoked by the
harness, but was absent from the index (**Criterion 5 – Consistency**).

---

### `harness/templates/kitchen_sink.md`

**What changed:** Added a deprecation warning comment directly above the `harmonious_color`
helper function signature, noting it must not be used in `build_scenes` or `scene_repair`
`scene_body` output and instructing use of hex strings instead.

**Why:** The kitchen sink already contained an override note at the top (line 13) and an explicit
"Wrong" example at line 48, but the helper signature at line 30 had no deprecation marker. An
agent scanning only the helper signatures section could conclude `harmonious_color` is safe to use
(**Criterion 1 – Specificity**, **Criterion 6 – Redundancy** reduction of contradiction).

---

### `harness/templates/visual_helpers.md`

**What changed:** Added a blockquote deprecation notice at the top of the "Enhanced Visual
Helpers" section, explicitly stating `harmonious_color` must not appear in `scene_body` output
because it returns RGBA float arrays (not Manim color strings), and directing agents to use hex
strings.

**Why:** `visual_helpers.md` said "Always include these functions in scene files" immediately
before showing `harmonious_color` without any caveat. Both `04_build_scenes/user.md` and
`06_scene_repair/system.md` ban `harmonious_color` with a "Wrong" label and TypeError explanation,
creating a direct contradiction (**Criterion 2 – Constraint Completeness**, **Criterion 6 –
Redundancy** of conflicting instructions).

---

### `harness/templates/phase_scenes.md`

**What changed:** Added an "Ownership note" blockquote at the start of the `build_scenes` process
list, clarifying that steps 1, 2, 5, 6, and 7 are script-owned and the agent's sole
responsibility is step 3 (filling in animation statements). Added a "Script-Owned – Not an Agent
Task" heading and blockquote to the Pre-Render Validation section.

**Why:** `phase_scenes.md` is injected as `{{scenes_doc}}` into the `scene_qc` system prompt and
also serves as pipeline reference material. The original text described scaffolding commands,
state mutations, and validation scripts in imperative language without indicating that these are
orchestrator responsibilities. This violated **Criterion 3 – Ownership Alignment** and could
cause agents to attempt to perform script-owned work.

## Deferred Recommendations

The following improvements were identified but require changes outside `harness/prompts/` and
`harness/templates/` and are therefore logged here rather than implemented:

1. **Manifest-driven validation in `harness/prompts.py`:** The `manifest.yaml` files define
   `validation` blocks (scene count ranges, required fields) that are not currently read by the
   harness at runtime. Implementing manifest-driven validation in `harness/prompts.py` or
   `harness/parser.py` would surface plan violations deterministically before model output is
   accepted. Requires changes to `harness/prompts.py` and `harness/parser.py`.

2. **Shared partial includes for repeated layout checklists:** The "Hard Layout Contract" in
   `04_build_scenes/user.md` and the "Mandatory Repair Validation Checklist" in
   `06_scene_repair/user.md` duplicate the same seven rules. A template include mechanism (e.g.,
   `{{layout_contract}}` resolved from `_shared/`) would remove this redundancy without manual
   synchronization. Requires changes to `harness/prompts.py` template rendering.

3. **Slot-helper scaffolding:** `DEVELOPMENT_GUIDELINES.md` calls for `play_in_slot` /
   `play_text_in_slot` helpers that own timing slot allocation. These helpers are referenced but
   not yet generated by `scripts/scaffold_scene.py`. Until they exist the prompts correctly guide
   agents to use `tracker.duration` fractions, but the aspirational deterministic timing contract
   cannot be fully enforced. Requires changes to `scripts/scaffold_scene.py` and
   `flaming_horse/scene_helpers.py`.

4. **`harmonious_color` removal from scaffold imports:** The function is imported in
   `harness/templates/manim_template.py.txt` and `flaming_horse/scene_helpers.py`. Removing or
   replacing it in those locations would eliminate the source of confusion entirely. Requires
   changes to `flaming_horse/scene_helpers.py` and `manim_template.py.txt`.

5. **`phase_scenes.md` process list cleanup:** Steps 1, 2, 5, 6, and 7 of the `build_scenes`
   process (state reads, scaffolding commands, state updates) should be removed from the template
   entirely since they describe script-owned work. Keeping them risks confusing future agents even
   with the ownership labels added in this PR. A focused PR can remove those steps after
   confirming no test or tooling reads them.
